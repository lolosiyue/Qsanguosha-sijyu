# 一人多控初版说明

## 概述

本次加入的是「一人多控 / Context Switching Dual-Control」初版实现。

核心目标：

- 允许主控者代替被控者处理需要客户端交互的请求
- 在非操作阶段同步被控者的手牌可见信息
- 保持原有协议结构，不改 `QSanProtocol` 底层解析
- 以 `objectName()` + `Controller_Name` Tag 建立控制关系
- 采用「基于请求的上下文切换」而不是常驻式 UI 视角切换

这套实现不是永久接管视角，而是：

1. 先建立控制关系与手牌可见性
2. 真正发生请求时，临时把客户端切到被控者视角
3. 请求结束后再切回主控者本尊

---

## 功能作用

### 1. 主控者代答交互请求

当 `Room::doRequest()` 的目标角色存在有效控制者时，服务器会把请求重定向给主控者处理。

适用场景包括：

- 出牌请求
- 打出/响应请求
- 选牌、选人、拼点、无懈等交互

### 2. 主控者可查看被控者手牌

建立控制关系后，服务器会把被控者手牌同步给主控者，使其能在非请求阶段看到被控者的手牌信息。

注意：

- 这里只是「可见性同步」
- 真正把手牌拉到 Dashboard 上操作，仍然只发生在 `doRequest()` 临时切视角期间

### 3. 断线、死亡、重连时尽量自动恢复

初版已覆盖以下保护：

- 请求结束后强制回切本尊视角
- 当前临时 `Self` 死亡或移除时，客户端强制 `setSelf(m_original_self)`
- 控制关系解除时，若目标为 `offline`，立即改为 `trust` 并重置 AI
- 主控者重连后，重新补发被控者的 `S_COMMAND_SET_KNOWN_CARDS`

---

## 当前调用方式

### C++

`Room` 新增了统一入口：

```cpp
room->setPlayerController(target, controller);
room->setPlayerController(target, nullptr); // 解除控制
```

### Lua

Lua 侧保留了薄封装：

```lua
setDualControl(room, controller, target)
```

该函数内部直接调用：

```lua
room:setPlayerController(target, controller)
```

## 核心生命周期

### A. 建立控制关系

调用 `Room::setPlayerController(target, controller)` 时：

1. 清理旧控制者的被控手牌缓存
2. 写入 `target` 的 `Controller_Name` Tag
3. 给新控制者补发 `S_COMMAND_SET_KNOWN_CARDS`

这里不会发送 `S_COMMAND_SWITCH_CONTEXT`。

### B. 请求驱动的临时切换

当 `Room::doRequest()` 发现 `player` 存在在线控制者时：

1. 向主控者发送 `S_COMMAND_SWITCH_CONTEXT`，切到被控者视角
2. 将请求实际发给主控者
3. 用 `m_dualControlReplyOwners` / `m_dualControlRequestTargets` 记录「实际应答者」与「原请求角色」映射
4. 由主控者提交回复，服务器再把结果归还原角色

### C. 请求结束后还原本尊

同步等待路径中，`doRequest()` 使用作用域守卫保证离开作用域时一定执行清理：

- 清除 dual-control request 状态
- 发送 `S_COMMAND_SWITCH_CONTEXT` 切回主控者本尊

也就是说，即使发生以下情况，也必须回切：

- 超时
- 断线
- 提前释放等待锁
- 中途 return

### D. 重连补发

`Room::marshal()` 在主控者重连时，会再次检查有哪些角色当前受其控制，并重新补发：

- `S_COMMAND_SET_KNOWN_CARDS`

避免主控者重连后切到被控者时手牌显示为空。

## 注意事項

### 1. 严禁在 `setPlayerController()` 中发送 `S_COMMAND_SWITCH_CONTEXT`

`setPlayerController()` 只能建立控制关系和同步可见性，不能在建立关系时强行切 UI。

原因：

- 这会破坏「请求驱动的上下文切换」生命周期
- 客户端会长期停在被控者视角
- 一旦之后服务器向主控者本人发起普通请求，UI 视角会错位
- 可能造成无法响应、流程卡死

### 2. `SWITCH_CONTEXT` 必须和 `doRequest()` 绑定

正确生命周期只能是：

1. 切到被控者
2. 发请求
3. 等回复
4. 切回本尊

不能把切换行为散落到建立关系、普通通知、重连同步等非请求路径上。

### 3. 客户端恢复本尊必须走 `setSelf()`

不能写成：

```cpp
Self = m_original_self;
```

必须写成：

```cpp
setSelf(m_original_self);
```

否则 `switch_control_context` 不会发出，Dashboard 不会重新 `bindPlayer()`，容易停在死亡/灰头像上。

### 4. 解除控制时要处理离线目标

`Room::setPlayerController(target, nullptr)` 的解除控制分支里，如果 `target` 当前是 `offline`，必须立即：

- 切成 `trust`
- `resetAI(target)`

否则轮到该角色回覆时，服务器会一直等一个不存在的客户端。

### 5. 重连时必须补发被控手牌

`Controller_Name` 保存在 `ServerPlayer` 上，重连不会自动丢失。

但客户端重连后手牌缓存会重建，所以必须在 `marshal()` 或等价重连同步流程里，重新把被控者手牌通过 `S_COMMAND_SET_KNOWN_CARDS` 发给主控者。

### 6. 作用域清理不能漏路径

只要 `doRequest()` 在本次请求里切过视角，就必须保证无论任何 return 路径都能回切本尊。

建议：

- 使用 RAII / 作用域守卫
- 或者保证所有 return 前都调用同一套清理逻辑

### 7. 兼容旧 `onsole` 逻辑时不要混用语义

当前实现仍保留旧 `onsole` 路径作为 fallback，但新的双控链路应优先走 `Controller_Name` + 请求映射。

不要把两套视角切换逻辑混成一个“常驻控制模式”。

### 8. 修改 SWIG 后必须重编译

本功能修改了：

- `swig/sanguosha.i`

因此修改后必须重新编译整个项目，确保 Lua Wrapper 与 C++ 声明一致。

## 初版边界

当前版本的边界如下：

- 这是「初版双控」，不是完整的常驻代理系统
- 非请求阶段只保证可见性，不主动切 Dashboard 操作视角
- 真正的控制操作只在请求期间发生
- 控制链支持解析与环路保护，但不鼓励设计复杂多段控制链

---

## 建议测试清单

1. 建立控制关系后：
   - 主控者能看到被控者手牌
   - 但 Dashboard 不会立即切过去
2. 被控者收到交互请求时：
   - 主控者视角临时切到被控者
   - 回复后自动切回本尊
3. 超时 / 信号释放 / 断线时：
   - 仍然会切回本尊
4. 被控者死亡或移除时：
   - 客户端能恢复回本尊视角
5. 主控者重连后：
   - 被控者手牌会重新补发
6. 解除控制且目标为 `offline` 时：
   - 目标立即进入 `trust`
   - 后续请求不会卡死
